1.1 権威 DNS を Route 53 へ移管した理由

Amplify Hosting は apex (hrtep.com) の解決先として CloudFront ディストリビューションへ向ける必要があるが、apex を CloudFront に向けるには ALIAS レコードが必須である。RFC 1912 で apex の CNAME は禁止されており、CloudFront IP は変動するため A レコード直書きも運用不可。お名前.com 標準 DNS には ALIAS 機能が存在しないため、AWS 標準で apex ALIAS を提供する Route 53 への移管が事実上の唯一解だった。

Route 53 への移管に伴い、お名前.com 側で稼働していた既存 19 件のレコード (MX 6 件 / apex SPF / DMARC / SendGrid DKIM / Google Workspace DKIM 等) はすべて値完全保持で Route 53 にコピー移植した。MX/SPF/DMARC の値は変更ゼロのため受信メールへの影響はない。apex / www の A レコードのみ ALIAS A に置換し Amplify CloudFront に解決する。移管完了以降、お名前.com Navi 側の DNS レコード設定は権威系統から切り離されており、参照されない (NS 委任先が Route 53 の 4 件であるため)。

Phase 3 完了後に Search Console プロパティを新オーナー kishigami@leadx-grp.com へ移管する目的で、apex TXT に google-site-verification=RCKEIS0h3XQdw1ZY8uvVUF2mTn3IZkquyNNpE-g4LPA を Route 53 配下の同一 TXT レコードに UPSERT で追加した (旧 SC 認証値および SPF を保持しつつ追加)。詳細は §5.2 参照。

04問合せフォーム送信経路

本節要旨 訪問者がフォーム入力 → reCAPTCHA で bot 排除 → Server Action が SES でメール送信 → HRtep 様 Gmail に着信。送信元は専用サブドメイン send.hrtep.com に分離し、apex の SPF/DMARC へ副作用を出さない設計。

4.1 動作フロー

  1. ユーザーがフォームに入力 → react-hook-form + zod によるクライアントバリデーション
  2. 送信ボタン押下 → grecaptcha.execute で reCAPTCHA v3 token 取得
  3. Server Action submitContact に POST
  4. サーバー側で Google Verify API により reCAPTCHA token 検証 (score < 0.5 で拒否、ヘッドレスブラウザ等の bot を排除)
  5. SESv2 Client で SendEmailCommand 実行
  6. HRtep 様 Gmail にメール着信、Reply-To で送信者本人へ直接返信可能

4.2 送信パラメータ

Reply-Toフォーム送信者の入力メールアドレス (Gmail で「返信」を押した際に送信者本人宛になる)

4.3 送信ドメイン分離 (なぜ send.hrtep.com か)

送信元を apex (hrtep.com) ではなくサブドメイン (send.hrtep.com) に分離した理由:

  • apex の SPF (v=spf1 include:_spf.onamae.ne.jp include:_spf.google.com ~all) に include:amazonses.com を追加せずに済む
  • apex の DMARC アラインメント設計 (p=none, rua=mori@hrtep.com) に影響しない
  • 受信メール (Google Workspace + onamae.ne.jp) への副作用ゼロ
  • DKIM CNAME 3 件と SPF TXT 1 件をすべて send.hrtep.com 配下に集約、apex 配下のレコードと区別が明瞭

4.4 SES サンドボックス継続

SES は Production Access を申請せず、サンドボックス継続運用とする。送信先は Verified Identity 登録済みの information@hrtep.com 1 件のみのため、サンドボックスの「Verified Identity 宛のみ送信可」制約に抵触しない。流量上限 (24h あたり 200 通) に対し想定流量は月 30 通程度で十分余裕がある。流量が急増し月 100 通超の兆候が出た場合に Production Access 申請を検討する。

5.2 apex TXT レコード一覧 (Route 53)

apex (hrtep.com) の TXT は Phase 3 移管時の SPF 1 件に加え、Google Search Console 所有権認証 TXT を旧 + 新オーナーの 2 値並列保持している。SPF を含む 3 値とも 1 つの TXT レコードに UPSERT で同居させる仕様のため、追加・削除時は list-resource-record-sets で現値全件を取得し、保持対象を含めて UPSERT する必要がある (DELETE 時は値完全一致が必須)。

※ send.hrtep.com 配下の SES SPF / DKIM 群、_dmarc.hrtep.com の DMARC、default._domainkey.hrtep.com / google._domainkey.hrtep.com の DKIM は別レコード (apex TXT とは別管理)。

MX レコード変更禁止 MX は priority 1 で mail74.onamae.ne.jp と Google Workspace が並列。受信実態 (どちらが実質的に受信しているか) は文書化されていない。Phase 4 着手前のテスト送信で受信ヘッダの Received: 行解析により実態を特定する (§7.2 参照)。実態確定前に MX を変更すると受信障害を起こすリスクがある。

06緊急時ロールバック

運用開始後に致命的な問題が発生した場合の旧 WordPress 環境への復帰手順。

本節要旨 発火条件 (HTTPS 不到達 / フォーム継続失敗 / 受信メール異変 / DNS 解決異常) のいずれかを満たしたら、お名前.com Navi で NS を「お名前.com の設定を利用」に戻す。Route 53 hosted zone は削除せず保持し、再切替に備える。受信メール異変は最優先発火条件。

6.1 即時ロールバック発火条件

  • https://hrtep.com または https://www.hrtep.com が HTTPS で表示できない (証明書エラー / 5xx 連続)
  • お問合せフォーム送信が継続的に失敗する
  • 受信メールに意図しない変更が発生した (最優先発火条件)
  • DNS 解決異常 (apex/www が長時間解決できない)

6.2 ロールバック手順

  1. お名前.com Navi にログイン (お名前 ID 93206297)
  2. ドメインプロテクションを一時解除 (SMS 認証発生時は HRtep 様協力)
  3. 「ドメイン機能一覧」→「ネームサーバー設定」→ hrtep.com → 「お名前.com の設定を利用」 (ns-rs1/2.gmoserver.jp) を選択 → 確定
  4. nslookup -type=NS hrtep.com 8.8.8.8 で旧 NS への切戻り確認
  5. 受信メール / Web 表示の復旧確認 (nslookup -type=MX hrtep.com 8.8.8.8 で 6 件すべて baseline 一致を確認)
  6. ドメインプロテクションを再設定

※ Route 53 hosted zone は削除しない。再切替時にそのまま再利用可能で、保持しても月課金は $0.50 のみ。

6.3 ロールバック後の伝播時間

NS の TTL は通常 172,800 秒 (2 日)。Phase 3 切替時に短縮しなかった場合、最大 2 日のキャッシュ残留が発生する。緊急時は NS 戻しを最優先で実施し、伝播待ちは並行で進める。.com TLD 権威レベルでは即時反映 (実測 1 分以内) されるが、世界中の resolver キャッシュが切り替わるまでにはタイムラグがある。

07Phase 4 — 旧 WordPress 解約

Phase 3 完了 (2026-05-04) から並行稼働期間 14 日以上を経た後に着手する旧環境解約フェーズ。受信メール経路の実態に応じて解約範囲が分岐する。

本節要旨 解約対象は RSプランと ml-cp.hrtep.com の A レコード。お名前メールは受信経路の実態によって解約可否が分岐するため、受信ヘッダ解析でケース (a)/(b)/(c) を判定した上で決定する。判定確実性が低い場合は「継続」(=安全側) を選択する。

7.1 着手前提条件 (5 件、すべて充足が必須)

  1. Phase 3 完了から 14 日以上経過 (2026-05-18 以降)
  2. HRtep 様 Gmail (information@hrtep.com) でテスト送信着信を確認済
  3. 受信メール経路の実態をケース判定済 (§7.2)
  4. 新サイトで実問合せ 1 件以上の送受信実績、または Search Console での新ドメイン認識完了
  5. 旧 WordPress データの最終バックアップ取得済 (§7.4-A)

7.4 実行手順

  3. wp-config.php をローカル保存 (DB 認証情報を含むため漏洩管理に注意)
  4. MySQL データベースを phpMyAdmin または wp-cli db export で SQL ダンプ取得

  3. 確認画面で内容確認、申請確定

  1. お名前.com Navi → ご利用中のサービス → お名前メール → 「解約申請」

※ TTL と Value は実レコード値と完全一致が必須 (Route 53 の DELETE 仕様)。事前に list-resource-record-sets で現値を確認のうえ実行する。

7.4-E. 解約後 30 日リテンション期間の検証

RSプラン解約後、お名前.com の規約により旧 WordPress 環境は 30 日間ホスティング側で保持される (即時削除されない)。この期間に以下を検証する。

  • 第三者が https://hrtep.com にアクセスして新サイトが表示されること
  • HRtep 様 Gmail で受信メールに欠落・遅延が発生していないこと (ケース (a) で解約した場合は特に重要)
  • Google Search Console で旧 URL からのリダイレクトが認識され、新 URL のインデックスが進行していること
  • ドメイン更新料金が永久無料特典消失により通常料金 (推定 1,408 円/年) に変動することを次回更新時請求書で確認

7.5 Phase 4 中止判定

Phase 4 中止条件
  • 並行稼働期間中に新サイトの HTTPS 不到達 / 5xx 多発 / フォーム送信エラー多発のいずれかが発生
  • HRtep 様 Gmail で受信メール欠落・遅延が観測される
  • 受信経路ケース判定 (§7.2) が確定不能

中止時は §6 のロールバック手順を即時実行し、Phase 4 は無期限延期。原因究明後に再判定する。

08保守時の注意事項・運用知見

本節要旨 MX・apex SPF/DMARC・ドメインプロテクション・他事業由来の AWS リソース・既存の SES 別 Identity の 5 項目は変更禁止対象。Amplify 環境変数は全置換仕様のため、変更時は 3 件すべて含めて指定した上で再ビルドのトリガが必須。Domain Association 再作成時は CloudFront FQDN が変動する場合があり、Route 53 ALIAS Target の手動 UPSERT を要する。

8.1 変更禁止項目

  • MX レコード: 受信メール経路に直結。値変更は受信障害の直接原因となる
  • apex SPF / DMARC: 既存メールインフラとアラインメントしている。送信ドメイン追加は send.hrtep.com サブドメイン側で行う
  • ドメインプロテクション: 常時 ON 維持。NS 変更等の保守時のみ一時解除し、作業完了後すぐ再設定
  • HRtep 様 AWS アカウント内の他事業由来リソース: 同 AWS アカウント内に人材プラットフォーム関連の RDS / VPC / EC2 / S3 / Global Accelerator が同居。本案件範囲外につき関与対象外
  • SES の hrtep.net Domain Identity / HRtep Configuration Set: 既存の他用途リソース。本案件で関与しない

8.2 コスト監視

  • AWS Budgets hrtep-monthly-1000jpy が月 $7 (約 1,000 円) 超過で tbnkin@gmail.com 通知。本案件の安定運用月額 (約 463 円・税込) に対し十分なバッファ
  • Cost Explorer での内訳確認は claude-code-operator IAM ユーザーに AWSBillingReadOnlyAccess + AWSCertificateManagerReadOnly インラインポリシーが付与済。サービス別フィルタで本案件のみ抽出可能
  • HRtep 様 AWS アカウント全体の月額には他事業由来の課金 (RDS/VPC/EC2/S3/Global Accelerator で月 約 $440) が含まれる。Budget 超過判定時は本案件由来か他事業由来かを切り分ける

8.3 Amplify のデプロイ

  • GitHub main ブランチへの push で自動ビルド・自動デプロイ
  • 環境変数を変更する場合、aws amplify update-app --environment-variables は全置換のため、既存の NEXT_PUBLIC_RECAPTCHA_SITE_KEY / RECAPTCHA_SECRET_KEY / CONTACT_EMAIL_TO をすべて含めて指定する
  • 環境変数変更後は aws amplify start-job --job-type RELEASE で再ビルドをトリガーする (build phase で .env.production に書き出されるため、再ビルドしないと反映されない)

8.4 SES の運用知見

  • DKIM 検証が FAILED 状態で停滞した場合、signing-enabled の toggle では再起動しない。delete-email-identity + create-email-identity による Identity 再生成で内部 polling が確実に再起動される。Easy DKIM の Tokens は AWS account 単位で決定論的に払い出されるため、再生成しても Route 53 CNAME の再投入は通常不要
  • 送信先 (information@hrtep.com) の Verified Identity 登録は SES サンドボックス継続の前提条件。HRtep 様 Gmail を変更する場合は、新アドレスを Verified Identity 登録 + Verification リンククリックが必要
  • bounce/complaint rate が想定 (5% / 0.1%) を超過すると CloudWatch Alarm 発火 → SNS Topic hrtep-cloudwatch-alarms 経由で通知

8.5 Free Tier 12 ヶ月終了済の影響

HRtep 様 AWS アカウントは新規作成から 12 ヶ月以上経過しており、AWS Free Tier (新規アカウント 12 ヶ月限定枠) は終了している。Amplify Hosting は SSR Compute / Build minutes / Data transfer のいずれかで月 約 320 円の微小課金が発生する。低トラフィックのコーポレートサイトでは月 数百円規模で安定する見込み。

8.6 Amplify Custom Domain と ACM の連動

Amplify Custom Domain Association を関連付けると Amplify が ACM 証明書を自動発行・自動関連付け・自動更新する (DNS validation 方式、Amplify Managed Certificate)。証明書更新の運用工数はゼロ、HRtep 様の検証メールクリック等は発生しない。証明書発行から DNS 反映までの 72 時間制約があり、関連付け実行から 72h 以内に検証 CNAME を Route 53 で解決可能にする必要がある。タイムアウトした場合は delete-domain-association + create-domain-association で再関連付け可能で、検証 CNAME 値は決定論的に同一値が再払い出される。

8.7 Amplify Custom Domain 関連付け再作成時の注意

Domain Association を delete-create で再作成すると Amplify CloudFront FQDN が変わる場合がある (例: d1r3kbuaudozwk.cloudfront.net → d3u3xuwk9ma2bd.cloudfront.net)。Route 53 上の apex/www の ALIAS Target は Amplify が自動更新しないため、新 FQDN への手動 UPSERT が必要。Amplify ステータス AWAITING_APP_CNAME はこの状態を示す。

8.8 NS 切替の伝播

NS 切替は権威 DNS (.com TLD レベル) では即時反映 (実測 1 分以内)。ただし世界中の resolver キャッシュは TTL 残留により伝播完了まで最大 2 日要する場合がある。ロールバック時は NS 戻し直後から正常応答が始まるが、最大 2 日のキャッシュ残留が並行する点に注意する。

09用語集

本節要旨 本書で頻出する技術用語のうち、運用引継ぎ・障害対応の場で 意味を取り違えると判断ミスにつながるものに絞って簡潔に定義する。
apex (zone apex)
サブドメインを伴わないドメイン本体 (hrtep.com のこと)。RFC 1912 により apex に CNAME を投入することは禁止。Amplify への apex 解決には ALIAS レコードが必須となるため Route 53 を採用した。
ALIAS レコード
Route 53 固有の拡張レコード型。CNAME と異なり apex に対しても投入でき、AWS リソース (CloudFront / ELB / S3 等) のターゲット FQDN を直接指定できる。お名前.com 標準 DNS には存在しないため、本案件で Route 53 へ移管した最大の理由。
権威 DNS (Authoritative DNS)
そのドメインのレコードを 正本として保持するサーバ。世界中のリゾルバはここに問い合わせる。本案件では Route 53 が権威 DNS、お名前.com はレジストラ (登録代行) のみ。
NS (Name Server) 切替
レジストラに登録するネームサーバ情報を Route 53 の 4 件に変更する操作。.com TLD レベルでは即時反映だが、世界中のリゾルバ・キャッシュへの伝播は最大 2 日要する場合がある。
MX レコード
そのドメイン宛のメールをどのメールサーバに配送するかを宣言するレコード。本案件で 6 件 (Priority 1 が onamae.ne.jp と Google Workspace、Priority 5 が Google の予備 4 件) を完全保持。変更時は即時に受信障害が発生する。
SPF / DKIM / DMARC
送信ドメイン認証の三点セット。SPF は送信元 IP の許可、DKIM は電子署名、DMARC はそれらが揃わなかった場合の扱い (p=none は観測のみ)。本案件では apex 側 (受信用) は完全保持し、送信用は send.hrtep.com サブドメイン側に分離。
送信ドメイン分離
送信専用にサブドメイン send.hrtep.com を立て、apex 側の認証設定に副作用を出さない設計。送信側でスパム判定上の問題が発生した場合でも受信側に波及しない安全策として機能する。
ACM (AWS Certificate Manager)
AWS の SSL/TLS 証明書発行・自動更新サービス。Amplify Managed Certificate は ACM をバックエンドに使い、DNS 検証 + 自動更新で運用工数ゼロ。
CloudFront
AWS の CDN (世界配信網)。Amplify Hosting は内部で CloudFront ディストリビューションを生成し、apex/www の ALIAS Target がそれを指す。FQDN は Domain Association 再作成時に変わる場合がある (§8.7 参照)。
Server Action
Next.js App Router のサーバ実行型関数。フォーム送信を API Route なしで直接 SSR から処理できる。本案件では submitContact (app/contact/actions.ts) がこれにあたる。
reCAPTCHA v3
Google のスコアベース bot 判定。チェックボックス UI を出さず、行動シグナルから 0.0–1.0 の score を返す。本案件は閾値 0.5 で拒否。
SES サンドボックス
SES 新規アカウントのデフォルト状態。送信先は Verified Identity 登録済アドレスのみ・24h あたり 200 通・1 秒あたり 1 通の制約。本案件は送信先 1 件 + 月 30 通想定で抵触しないため継続運用。
Configuration Set
SES 送信ジョブにメタデータ (bounce/complaint 通知先・統計等) を紐付ける設定束。本案件では hrtep-default を使用し、CloudWatch Alarm 連携の起点となる。
bounce / complaint
bounce は不達 (存在しないアドレス等)、complaint は受信者からのスパム報告。SES では発生率に閾値 (5% / 0.1%) があり、超過すると送信権が制限されうる。本案件は CloudWatch Alarm で監視。
ドメインプロテクション
お名前.com のオプション。NS 変更・移管等の重要操作時に SMS 認証を要求するセキュリティ機能。本案件では常時 ON、保守時のみ一時解除する運用。
RSプラン
お名前.com のレンタルサーバ + ドメイン更新永久無料セット。本案件で旧 WordPress を稼働させていたサーバ契約を指す。Phase 4 で解約予定 (永久無料特典は消失し、ドメイン更新料金が新たに発生)。
TTL (Time To Live)
DNS キャッシュの保持秒数。NS の TTL は通常 172,800 秒 (2 日)。短縮しなかった場合、ロールバック後も最大 2 日のキャッシュ残留が発生する点に注意。
CNAME 検証 (DNS validation)
ACM 証明書発行時、ドメイン所有を確認するために特定の CNAME レコードを公開する方式。Amplify Custom Domain は自動で Route 53 に投入する。NS 切替が完了していないと Amplify 側ポーリングが空振りして Domain Association が FAILED になる (Phase 3 で実際に発生)。
