04Web ホスティング先 — AWS Amplify Hosting

HRtep 様の既存 AWS ご利用環境との親和性が高く、4 つの観点で他候補を上回る。

結論Web ホスティング先は AWS Amplify Hosting（東京リージョン / ap-northeast-1）。月額は実質 0〜750 円、Free Tier 内で完結する。既存 AWS 環境との一体的な運用が可能であり、Next.js 16 を amplify.yml 経由で動作させる。

4.1 推奨理由（4 つの優位性）

① 既存 AWS 環境との統合

HRtep 様の既存 AWS 環境（アカウント 688518755365）と同一アカウント内にリソースを配置することで、請求・IAM・運用知見を一元化できる。新たなクラウドベンダーを追加することなく、既にご評価いただいている AWS のセキュリティ・契約条件をそのまま適用可能。

② ランニングコストが実質 0 円

Amplify Hosting の Free Tier（ビルド 1,000 分・ストレージ 5 GB・データ転送 15 GB・SSR リクエスト 50 万件 ／月）に対し、HRtep 様サイトの想定アクセス規模では数 % 以下の消費に留まる見込み。Vercel Pro / Netlify Pro（年額 36,000 円相当）と比較して大幅に安価。

③ 旧 WordPress 解約による費用相殺

移行完了後（Phase 4）に お名前.com の WordPress ホスティング契約を解約することで、新ホスティング費用との純差額がプラスとなる。加えて、WordPress のプラグイン更新・PHP/MySQL バージョン管理・脆弱性対応に伴う運用負荷も発生しない。

④ Next.js 16 への対応

Next.js の SSR・Server Action・ISR を amplify.yml 経由でフルサポート。問合せフォームの送信処理を Server Action（app/contact/actions.ts）で完結できるため、別途 API 用のインスタンスを用意する必要がない。HTTPS 証明書（ACM）も自動発行・自動更新となり、証明書更新の運用工数は発生しない。

6.2 Amazon SES を採用する根拠

  • ベンダー統合: Web ホスティング（Amplify）と同一クラウド内で完結し、請求・IAM・監査ログ・障害対応窓口が AWS に集約
  • セキュリティ評価の継承: 既に AWS のセキュリティ・契約条件をご評価いただいている前提を活用、新たな業者選定不要
  • コスト: サンドボックスは送信料無料、Production 段階でも月 1,000 通で年額 18 円程度
  • 運用: Amplify の IAM ロール（hrtep-amplify-compute-role）から直接 SES を呼び出すため、API キーの個別管理が不要

6.3 サブドメイン送信元構成（送信ドメイン分離）

採用構成: noreply@send.hrtep.com 送信ドメインを apex の hrtep.com から分離し、サブドメイン send.hrtep.com として SES Domain Identity を登録。これにより:
・apex の SPF（v=spf1 include:_spf.onamae.ne.jp include:_spf.google.com ~all）に include:amazonses.com を追加する必要がない（apex 無変更）
・apex の DMARC アラインメント設計に影響なし
・DKIM CNAME 3 件（AWS Easy DKIM、RSA 2048bit）も send.hrtep.com 配下に集約され、apex 配下のレコード群と区別が明瞭
受信メール（Google Workspace + onamae.ne.jp）への影響はゼロ。

DKIM 認証は §11 Step A-2 でも未完了で、§12 Step 2（行 1〜4）追加と DNS 伝播後に AWS が自動検証して SUCCESS に。本書の手順通り「§11 Step A-4 → §12（72h 以内）→ §13」の順で進めば、新サイト切替時には DKIM SUCCESS 済み状態となり、フォーム送信は初回から稼働する。

14設計判断と教訓

本案件で行った設計判断の根拠と、想定される失敗パターンを、類似プロジェクト・引継ぎ先での再現に資する形で記録する。

狙い「なぜこの構成にしたのか」「どのような失敗が想定されるか」を言語化することで、本構成を再現しやすくし、他プロジェクトへの転用判断を容易にする。

14.1 なぜ AWS Amplify Hosting か（vs 他選択肢）

Lesson 01 / Hosting Decision

既存環境との統合 > 単体性能

Vercel / Netlify は Next.js への対応が最も洗練されているが、既に AWS を使っている顧客に対しては「新規ベンダー追加コスト」（契約・経理・セキュリティ評価・運用知見の分散）が想定以上に重い。Cloudflare Pages は安いが Workers ランタイム制約があり、AWS 環境からの分散も生じる。

AWS Amplify は Next.js 公式 Adapter の対応に遅延が出る場合があるが、amplify.yml でビルド定義を自前管理すれば回避可能。既存 AWS 環境内で Free Tier に収まるコーポレートサイトであれば、ベンダー統合のメリットが他候補の単体性能を上回る。

TAKEAWAY候補比較は「単体性能」だけでなく「顧客の既存環境との統合性」を主軸に評価する。新規ベンダー追加のコストは月額単価に現れない

14.2 なぜ送信ドメインを send.hrtep.com に分離するのか

Lesson 02 / Email Domain Separation

apex の SPF/DMARC を編集すると受信に副作用が出る

apex の hrtep.com は既存 SPF（include:_spf.onamae.ne.jp include:_spf.google.com）と DMARC（p=none, rua=mori@hrtep.com）が稼働中。ここに SES 送信を追加するため include:amazonses.com を apex SPF に追記すると、既存メール送信の SPF 評価が変わり、Google Workspace の DMARC アラインメントが再設計対象になる。

サブドメイン send.hrtep.com を切り出して SES Domain Identity を登録し、DKIM CNAME 3 件と SPF TXT 1 件をその配下に集約することで、apex は完全無変更を維持。受信メール（MX）への副作用がゼロ。

TAKEAWAY既存メールインフラのある apex に手を入れる前に、送信専用サブドメインへの分離を検討する。受信影響ゼロの最短経路

14.3 なぜ SES サンドボックスで足りるのか（流量設計）

Lesson 03 / SES Sandbox Sufficiency

流量実績を見てサンドボックス内で完結するか先に判断する

SES のデフォルト動作はサンドボックス（24h 200 通制限・Verified Identity 宛のみ）。多くのプロジェクトでは「いずれ Production Access が必要」と仮定して即申請するが、本案件は実流量月 30 通以下（既存 WP + CF7 過去 12 ヶ月実績）。サンドボックス制限の 0.5% 以下に収まるため申請不要。

送信先（To）も Verified Identity 登録が必要だが、本案件は固定 1 件（info@hrtep.com）のため Verification リンククリック 1 回で済む。申請しないことで AWS Trust & Safety 審査（24h〜数日）の待機が不要になり、リードタイムが短縮。

流量増加時の Production Access 申請文書（docs/ses-production-access-request.md）はテンプレ化済で、必要時に即提出可能。

TAKEAWAYSES Production Access は「必要な時に必要なだけ」。流量実績を確認せず即申請するのは過剰

14.4 なぜ MX に触らないか

Lesson 04 / MX Untouchability

既存メールインフラの全貌を理解せずに MX を変更すると受信障害を起こす

本案件の MX 6 件は Google Workspace 5 件 + mail74.onamae.ne.jp 1 件の並列構成。priority 1 で並列に存在し、受信実態が文書化されていない。考えられるパターン:

  • (a) mail74 がプライマリ、転送経由で Google Workspace
  • (b) 過去設定の残骸で実態は Google Workspace のみ
  • (c) 両方並列受信

Phase 3 では MX を一切触らず、Phase 4（旧 WP 解約）前の最終テスト（§13.2 #4）で受信ヘッダの Received: 行解析により実態確定する。実態が判明する前にお名前メールプランを解約すると、(a)/(c) の場合に受信が突然止まる事故になる。

TAKEAWAYMX は「動いている」=「触らない」が大原則。実態が文書化されていない場合は、変更ではなくテスト送信での経路特定を先に

14.5 72 時間制約の意味（ACM 仕様）

Lesson 05 / ACM 72h Timeout

DNS validation は払い出しから 72h 以内に DNS 反映が必須

AWS Certificate Manager の DNS validation は CNAME 発行から 72 時間以内に DNS が反映されないとリクエスト状態が VALIDATION_TIMED_OUT に遷移してリクエスト無効化。Amplify Custom Domain は内部で ACM 証明書を発行するため同じ制約が適用。

タイムアウトしても再関連付け（delete-domain-association → create-domain-association）で再払い出し可能だが、検証 CNAME 値は別の値になるので DNS 設定書の行 5・6・7 を全て書き直す必要がある。実用上の許容遅延は DNS 伝播時間（最大 24h）を引いた 48h、安全マージン込みで「同日内に DNS 設定」を運用ルールにする。

TAKEAWAYAWS の検証 CNAME 払い出しは「期限が起算される」操作。事前準備が完了してから払い出しを実行する

14.6 DKIM/SPF/DMARC の役割分担

Lesson 06 / Email Auth Architecture

3 つは独立して機能、サブドメイン分離で影響範囲を制御

DKIM: 送信ドメインの真正性をデジタル署名で証明（送信者がドメインの持ち主であることを示す。本案件は send.hrtep.com 配下に CNAME 3 件、AWS Easy DKIM RSA 2048bit）。

SPF: 送信元 IP がドメインから許可されているかを宣言（メールサーバーの IP リストを TXT で公開。本案件は send.hrtep.com に v=spf1 include:amazonses.com ~all）。

DMARC: DKIM/SPF の検証失敗時の挙動指定（quarantine/reject/none）+ レポート送信先（rua）。本案件は apex の DMARC を無変更（p=none, fo=1, rua=mori@hrtep.com）。

3 つは独立に機能するため、サブドメイン分離は DKIM/SPF の影響範囲をコントロールする手段。apex DMARC のアラインメント設計を再検討せずに済む。

TAKEAWAYメール認証 3 兄弟（DKIM/SPF/DMARC）は独立。「変えなくて済むものは変えない」設計が安全

14.7 お名前.com の DNS 管理形態の見極め（Navi vs RS plan UI）

Lesson 07 / Onamae DNS UI Selection

RS plan 利用ドメインは RS plan コントロールパネルの DNS UI が最良経路

お名前.com には DNS を編集できる UI が複数存在し、ドメインの契約形態で利用可否が分かれる:

  • お名前.com Navi 「ドメインDNS設定」 → 「DNSレコード設定」: NS が 01-04.dnsv.jp 必須。利用時に NS 移管チェックボックスで切替可能だが、移管後は dnsv.jp ゾーンが空のため 既存の MX/SPF/DMARC が消失する致命的リスクがある（実機検証で発覚）
  • RS plan コントロールパネル → ドメイン → DNS: NS = ns-rs1/2.gmoserver.jp（RS plan bundled）のドメインで利用可。既存全レコードが UI に表示・編集可、新規追加・削除・変更すべて対応。NS 移管不要。本案件で採用（Option Y）
  • AWS Route53 等の外部 DNS: NS を Route53 NS に移管する大移行。本案件では不採用（コスト + 手間 + 既存最大限活用方針）

本案件で実機検証 7 ステップを経て判明した事実:

  • HRtep 様契約の hrtep.com は RS plan bundled NS（ns-rs1/2.gmoserver.jp）→ Navi の DNSレコード設定メニューを使うには NS 移管が必要だが、dnsv.jp ゾーンが空のため危険
  • RS plan コントロールパネルには独立した DNS 編集 UI があり、既存 19 件レコード（SendGrid/Google MX/SPF/DMARC/DKIM 等）を完全に保持しつつ新規追加・既存変更・削除が可能
  • RS plan UI の安全機構: 変更時「上記の注意事項を確認しました」チェックボックス必須、UPSERT 動作、削除前の確認ダイアログ
  • 「初期レコード」マーク付きレコードでも値・TTL は変更可能（タイプ・ホスト名は変更不可）
  • ホスト名末尾の .hrtep.com 自動付加、TTL 範囲 300〜604800、TYPE プルダウン A/AAAA/CNAME/MX/TXT/SRV 対応
TAKEAWAY「お名前.com で DNS を編集する」と一括りにせず、ドメインの契約形態（RS plan / dnsv.jp NS / 共用サーバー / VPS / 他社外部）で UI と制約が異なることを最初に把握する。RS plan ユーザーは必ず RS plan UI を使う（Navi 経由は NS 移管リスクで危険）

14.8 AWS Amplify Custom Domain と ACM の連動

Lesson 08 / Amplify-ACM Linkage

Custom Domain 関連付けで ACM 証明書発行は自動化される

Custom Domain を関連付けると Amplify が内部で:

  1. ACM 証明書発行リクエストを作成（DNS validation モード）
  2. 検証 CNAME と Custom Domain ALIAS（apex は A）を返す
  3. DNS 反映後、ACM が CNAME を検証 → 証明書発行
  4. Amplify が証明書を自動関連付け
  5. その後の更新（90 日）も AWS が自動実施

全工程は数時間〜24h 程度（DNS 伝播時間に依存）。証明書更新の運用工数はゼロ。

TAKEAWAYAmplify Custom Domain = ACM 証明書管理の自動化。証明書更新カレンダーや監視は不要

14.9 典型的な失敗パターンと予防策

Lesson 09 / Pitfalls

本案件で予防した（または予防すべき）失敗パターン

  • (a) 順序逆転: Amplify Custom Domain 関連付け前に DNS を変更してしまう → ACM 検証 CNAME が分からないまま切替が始まり HTTPS 表示不可。順序は「AWS 事前準備 → DNS 設定」固定
  • (b) 過剰申請: 不要なケースで SES Production Access を申請 → AWS から「サンドボックスで運用可能なユースケースですね」と却下。流量根拠を明示すれば承認されるが、そもそも申請が必要かを先に判断
  • (c) apex SPF 編集: include:amazonses.com を apex SPF に追記 → 既存 Google Workspace の SPF アラインメントが壊れて受信側で SPF FAIL の可能性。サブドメイン分離で回避
  • (d) MX 変更: 受信メール停止の最重大リスク。本計画では一切触らない
  • (e) 警告無視できる事象を止める: www CNAME 追加時の「A レコード残存」警告は確認画面で同時コミットされるので無視して OK。止めると 2 段階作業の意義がなくなる
  • (f) DKIM CNAME を apex 配下に置く: AWS が apex を SES Identity として要求してしまう。サブドメイン送信元構成が不可能になる
  • (g) ACM 検証 CNAME 払い出し後の長期放置: 72h タイムアウトで再関連付けが必要。事前準備が完了してから払い出すルール
  • (h) ホスト名入力欄に .hrtep.com を含める: お名前.com Navi は自動付加するため、含めると xxx.hrtep.com.hrtep.com になる
  • (i) DNS 反映前に SES サンドボックス解除を申請: DKIM Status が PENDING のまま申請しても AWS から「DKIM 認証完了後に再申請を」と却下される
  • (j) DNS 管理形態を実機確認せずに作業着手: お名前.com には複数の DNS 編集 UI（Navi DNSレコード設定 / RS plan コントロールパネル / 外部 DNS）があり、契約形態で利用可否と制約が異なる。本案件では Navi DNSレコード設定で「8 件追加 + NS 移管」を計画していたが、dnsv.jp ゾーンが空であることが実機検証で判明し既存 MX/SPF/DMARC 消失の致命的リスクが発覚 → RS plan コントロールパネル UI 採用に Pivot。計画段階では実 UI と既存ゾーン状態を最初に確認する
  • (k) 同一お名前.com アカウント内の他ドメイン（hrtep.net）リソースに触れる: 既存の hrtep.net は別の業務システム本番稼働中。本案件は hrtep.com のみが対象、hrtep.net の DNS / SES / Amplify には一切触れない
  • (l) Amplify 環境変数 CONTACT_EMAIL_TO を設定し忘れる: app/contact/actions.ts は未設定時に「メール送信スキップ + 受付完了表示」のサイレント失敗分岐に入る。ユーザー画面では「お問合せを受け付けました」と表示されるが実メール送信されない。Phase 3 着手前に必ず Amplify App level 環境変数で CONTACT_EMAIL_TO=info@hrtep.com 設定 + 再ビルド（§11.5 Step A-5）
  • (m) ドメインプロテクションの保護対象を実機で検証せずに作業: お名前.com 公式は「DNS 関連機能の設定」を保護対象としているが、RS plan コントロールパネル経由の DNS 編集が含まれるかは未明文化。最初の 1 件追加コミット時に追加認証が出るかで判定する運用が必要
TAKEAWAY失敗パターンの多くは「順序」と「範囲」のミス。事前計画で順序を固定し、変更範囲を最小化する

14.10 本案件で採用した設計原則（再利用可能）

  • 変更範囲の最小化: 受信メール（MX/apex SPF/DMARC）は触らない、サブドメイン分離で apex 無変更
  • 既存環境との統合性優先: 顧客の既存クラウド環境内で完結する選択を最優先
  • 流量実績ドリブンの設計: SES サンドボックス vs Production の判断は仮定ではなく実流量で
  • 72h 制約の運用ルール化: AWS 側払い出しから DNS 反映まで「同日内」を目標、ACM タイムアウトで手戻りしない
  • 段階的切替（4 Phase）: AWS 構築 → SES 構築 → DNS 切替 → 旧 WP 解約。各 Phase で動作確認、問題発生時は前 Phase に戻れる
  • ロールバック手順の事前明示: 切替前にロールバック発火条件と手順を文書化、緊急時に判断で迷わない
  • 並行稼働期間 14 日: 緊急ロールバック猶予と Search Console 再認識を兼ねる
