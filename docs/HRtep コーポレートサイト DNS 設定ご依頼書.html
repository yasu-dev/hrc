変更の趣旨（3 つの目的）

  • ① お問合せフォームの送信メールに DKIM 認証を追加：Amazon SES で送信するため、新規サブドメイン send.hrtep.com 配下に DKIM CNAME 3 件 + SPF TXT 1 件を追加します（行 1〜4）
  • ② HTTPS 化のための ACM 証明書検証：apex / www 用に ACM 検証 CNAME 2 件を追加します（行 5・行 6）
  • ③ サイト本体の置き場所を切替：hrtep.com apex / www の値を、現行 WP の IP から Amplify が払い出す値に変更します（行 7・行 8、最後に実施）

※ 既存の MX（受信メール）・apex SPF・DMARC・その他のレコードには一切手を加えません（第 1 章末尾「絶対に変更しないでください」を参照）。
※ 行 5・6・7 の VALUE は、LeadX 社が依頼送付直前に Amplify から払い出される確定値を表に記入したうえでお渡しします。行 1〜4・行 8 は固定値です。

お名前.com Navi での操作手順

  1. お名前.com Navi にログイン
  2. 「ドメイン」 → 「DNS設定」
  3. 対象ドメイン hrtep.com の「DNS設定 / 転送設定」を選択
  4. 「DNSレコード設定を利用する」を選択して「次へ」
  5. 上記 8 件を以下の要領で操作
    • 1〜3（DKIM CNAME 追加）: ホスト名欄に xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx._domainkey.send の部分のみ入力（末尾の .hrtep.com は自動付加）／ TYPE: CNAME ／ VALUE: 表の値
    • 4（SPF TXT 追加）: ホスト名欄に send のみ入力／ TYPE: TXT ／ VALUE: v=spf1 include:amazonses.com ~all
    • 5〜6（ACM 検証 CNAME 追加）: ホスト名欄・VALUE ともに依頼書記載のとおり入力／ TYPE: CNAME／ ホスト名は末尾の .hrtep.com を除いた部分のみ入力（自動付加されます）
    • 7（A レコード変更）: ホスト名欄を空欄（apex 用）／ 既存の VALUE 118.27.122.212 を、依頼書記載の Amplify 確定値に置換
    • 8（www CNAME 変更）: ホスト名 www ／ 既存値を main.d3561ojkga3nyn.amplifyapp.com に置換（TYPE が A の場合は CNAME に変更）
  6. 8 件分すべて操作後「確認画面へ進む」 → 内容確認の上「設定する」を押下
絶対に変更しないでください 以下のレコードに触れると受信メール・ドメイン認証が壊れます。本作業の対象は前掲 8 件のみです。
  • hrtep.com の MX レコード（Google Workspace 等 6 件）
  • hrtep.com apex の SPF レコード（apex の TXT、v=spf1 include:_spf.onamae.ne.jp include:_spf.google.com ~all）
  • _dmarc.hrtep.com の DMARC レコード（TXT）
  • その他、本書面に列挙されていない既存のあらゆるレコード
受信メールへの影響なし 1〜6 は新規追加（DKIM CNAME 3 + SPF TXT 1 + ACM 検証 CNAME 2）、7〜8 は apex / www の値置換です。MX・apex SPF・DMARC レコードには一切手を加えません。Google Workspace ご利用の受信メールには影響しません。

切替直後の一時的なフォーム送信不通について

DNS 反映待ち中（最大 24 時間）の留意点 安井様の DNS 設定後、DKIM 認証が完了するまで（数時間〜最大 24 時間）の間、新サイト hrtep.com でのお問合せフォーム送信時にメール送信が一時的に失敗する可能性があります。
  • DKIM 認証は DNS 反映後に AWS が自動完了する処理。手動操作は不要
  • DKIM 反映待ち期間に新サイトへ送信されたフォーム入力が影響対象
  • DKIM 完了後は自動で正常稼働。LeadX 社が反映状況を監視します

もしものときのロールバック

切替後に問題（HTTPS 表示不可・受信メール影響など）が発生した場合は、安井様にて A レコードを元の値（118.27.122.212）に戻すだけで旧サイトに復帰します。DNS の TTL は 1 時間設定なので、最長 1 時間で完全復旧。
特に 受信メール（MX レコード）に意図せず変更が起きた場合は最優先でロールバックとする運用です。
